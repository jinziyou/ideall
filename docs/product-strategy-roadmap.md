# ideall 产品定位与实施路线图

> 状态：现行产品方向与执行基线
> 更新：2026-07-17
> 适用范围：产品定位、体验优先级、跨模块实施与版本验收

## 1. 决策摘要

ideall 保留“个人信息终端”作为长期品类愿景，对外定位收敛为：

> **一个本地优先的浏览与个人信息工作台：浏览、收集、整理，再让 AI 基于用户明确授权的资料检索、创作和行动。**

产品不以“拥有最多功能”为目标，也不把浏览器、笔记、Agent、应用启动器、社区和开发工具作为六条并列产品线。它们必须服务同一条主循环：

```text
浏览 / 发现 → 捕获 → 本地整理 → 检索与重访 → AI 综合 / 行动 → 形成个人产物
```

对应的能力层级为：

1. **浏览优先**是高频入口和产品楔子。
2. **本地个人数据**是用户资产与长期壁垒。
3. **检索与重访**决定资产是否真正有用。
4. **AI**是操作这些资产的环境层，不是孤立聊天页。
5. **轻量创作**承接阅读和 AI 的结果。
6. **资源、应用、信息与社区连接**是输入和输出通道，不单独争夺主心智。

## 2. 目标用户与核心任务

### 2.1 首要用户

首个稳定版本面向高信息密度的个人用户：

- 每天跨网页、文档、文件和 AI 工具研究或学习；
- 经常保存链接，却难以再次找到或利用；
- 希望资料主要由自己控制，并能离线访问；
- 愿意使用快捷键、标签页和 Agent，但不想维护复杂知识库；
- 典型角色包括独立开发者、研究者、学生、分析师、创作者和重度信息消费者。

### 2.2 暂不作为首要用户

- 以多人实时协作为核心的大型团队；
- 只需要极简待办或纯 Markdown 写作的用户；
- 需要完整企业权限、审批和项目管理的组织；
- 以社交内容消费为唯一需求的用户。

### 2.3 Jobs to be Done

核心任务按优先级排列：

1. 当我看到有价值的网页或资料时，能在两秒内保存，并保留来源与可检索信息。
2. 当我稍后需要某条信息时，能按记得的任意片段找到它，而不是重新上网搜索。
3. 当我需要理解、比较或产出时，能明确选择资料交给 AI，并知道 AI 使用了什么。
4. 当 AI 或阅读产生结论时，能立即沉淀为笔记、书签或后续任务。
5. 当资料位于外部应用或服务时，能在授权边界内连接、打开或行动。

## 3. 产品原则

### 3.1 本地拥有

- 用户创建和捕获的核心数据默认先落本机；离线时仍可浏览、检索和编辑。
- 联网、模型、同步、发布和外部连接分别说明接收方、数据范围与触发动作。
- 导出、备份与迁移是核心能力，不作为高级功能隐藏。

### 3.2 浏览不是终点

- 所有内容视图都应提供一致的“保存到我的”动作。
- 保存后的对象必须能够被搜索、引用、重新打开和交给 AI。
- “打开外链”只是最低能力；捕获、整理和再利用才是产品价值。

### 3.3 AI 可见、可控、可回写

- 默认只给 AI 最小必要上下文；正文读取需要用户明确选择或授权。
- 对话区展示当前上下文来源，写操作展示目标和结果。
- AI 的有价值产物必须能够回写为普通文件对象，脱离对话仍可使用。

### 3.4 轻创作而非全能创作套件

- 优先支持笔记、摘录、列表、代码块、引用和轻量发布。
- 不为追平完整文档、白板或项目管理产品而膨胀编辑器。
- 创作功能必须缩短“信息 → 观点/产物”的路径。

### 3.5 连接必须产生动作

- 应用、MCP、嵌入页和社区连接需要映射为可理解的资源或动作。
- 只展示入口、卡片或图标而没有捕获/读取/写回能力的连接不进入核心宣传。

## 4. 核心体验蓝图

### 4.1 浏览与发现

统一承接以下来源：

- 内嵌浏览器与外部链接；
- 新闻和社区内容；
- 本机文件、已安装应用和第三方 FileSystem；
- 搜索、关注源与用户主动导入。

所有来源至少提供：打开、复制链接、保存到我的；支持正文的来源进一步提供摘录、总结和引用。

### 4.2 捕获

捕获分三档，避免每次都弹复杂表单：

| 动作 | 默认结果 | 适用场景 |
| --- | --- | --- |
| 快速收藏 | URL、标题、摘要、favicon、捕获时间 | 稍后阅读与重访 |
| 保存摘录 | 选中文本、来源、页面标题、定位信息 | 研究证据与引用 |
| 保存快照 | 可检索正文、来源元数据、可选资源文件 | 防失效与离线阅读 |

首期只要求“快速收藏”形成稳定闭环；摘录和快照在后续版本加入。

### 4.3 整理

- 收藏夹和页面树提供显式归档位置；标签用于横向分类。
- 来源、捕获时间、内容类型和处理状态由系统维护，不要求用户手工输入。
- 提供“收件箱”视图承接未整理捕获，避免保存动作被分类选择阻塞。
- 允许 AI 建议标签和归档位置，但默认由用户确认。

### 4.4 检索与重访

检索按渐进路线实现：

1. 标题、标签、URL、描述和正文全文匹配；
2. 统一结果摘要、命中片段与类型过滤；
3. 本地增量索引，避免每次查询扫描完整正文；
4. 可选本地语义索引，并与关键词结果混排；
5. AI 基于用户选择的结果集回答，逐项给出来源。

### 4.5 AI 综合与行动

AI 的三种典型入口：

- **当前内容**：解释、总结、翻译、提取行动项；
- **选定资料**：比较多个网页、笔记或文件；
- **个人资料库**：在授权范围内检索后回答或执行。

每次运行至少呈现：当前工作区、模型、上下文对象、可用工具和写入目标。删除、发布、外部提交等高影响动作继续要求明确意图或确认。

### 4.6 产物与连接

AI 或阅读结果可以转为：

- 新笔记或现有笔记的引用块；
- 书签描述、标签或收藏夹归档；
- Agent 任务；
- 社区发布草稿；
- 外部应用/MCP 动作。

## 5. 信息架构

现有“我的 / 活动 / 浏览 / 应用 / 设置”五分区保持稳定，但产品解释调整为：

| 分区 | 用户问题 | 主要内容 |
| --- | --- | --- |
| 我的 | 我保存和创建了什么？ | 收件箱、关注、书签、资源、文件 |
| 活动 | 正在进行什么？ | AI 空间、任务、最近动作、删除 |
| 浏览 | 我现在要看什么？ | 新闻、社区、浏览器 |
| 应用 | 我可以调用什么？ | 搜索、本机应用、授权连接 |
| 设置 | 数据如何被处理？ | 本地数据、同步、外观、AI、权限 |

工作区继续作为显示模式存在：文件工作区是默认体验；音频和开发工作区属于面向特定用户的可选增强，不进入首屏价值叙述。

## 6. 数据与检索设计

### 6.1 当前可复用对象

现有 `note / bookmark / folder / file / feed / thread` Node 足以承载首期闭环，不为快速收藏新增第二套实体。

### 6.2 后续捕获元数据

在不破坏稳定身份的前提下，后续为捕获对象统一补充：

- `sourceUrl`、`sourceTitle`、`sourceKind`；
- `capturedAt`、`lastVisitedAt`；
- `contentHash`、`snapshotVersion`；
- `excerpt`、`author`、`publishedAt`；
- 引用定位与来源链。

字段应位于 Node 的受控 `meta` 或专用内容协议中，并提供版本迁移与导出格式。

### 6.3 检索边界

- UI 全文检索可以读取本机正文；不得因索引方便而扩大 Agent 默认权限。
- Agent 的资料库检索返回来源摘要和候选项；读取完整私密正文仍经过现有 consent gate。
- 索引只保存完成搜索所需的派生文本；敏感正文索引不得上传到默认后端。

## 7. 分阶段路线图

### R0：闭环起点——捕获与全文查找

目标：让浏览产生本地资产，并能再次找到。

- [x] 浏览器可读取当前页 URL、标题和正文。
- [x] 浏览器当前页一键保存到书签，自动生成摘要并去重。
- [x] 全局搜索覆盖标题、标签、URL、描述、关注条件、笔记正文和对话内容。
- [x] 搜索结果展示正文或元数据命中片段。
- [x] 新闻、社区和普通外链统一使用同一捕获动作。
- [x] 为首次捕获增加非阻塞的新手引导。

验收：用户从打开网页到保存不超过一次点击；随后输入标题之外的已保存摘要片段也能找到该对象。

### R1：可靠捕获——摘录、快照与收件箱

目标：网页失效或来源复杂时，个人资产仍然可靠。

- [x] 选中文本保存为带来源引用的摘录；页面选区为空时拒绝，按来源与文本去重。
- [x] 浏览器可选保存有界正文快照，以普通笔记离线阅读和编辑。
- [x] “我的 / 收件箱”集中处理未分类捕获，归档只移除收件箱标签，不复制或移动源对象。
- [x] 网页快照和摘录统一记录原始来源、捕获时间，并按 canonical URL 去重。
- [x] 收件箱统一导入浏览器书签 HTML、普通 HTML、PDF 与图片；书签去重，文件保留原始二进制。
- [x] 桌面系统投递入口：`ideall://capture` 深链接、单实例转发、文件“打开方式”和窗口拖放统一进入收件箱。
- [x] 移动端共用 `ideall://capture` URL 投递协议；冷启动与运行中深链接走同一接收链。
- [ ] Android/iOS 原生分享扩展，把系统分享面板中的图片与文件流转交现有投递协议。
- [x] 本地增量全文索引、v18 索引 schema 迁移、损坏回退和设置页手动重建工具（当前数据库 schema v20）。

验收：捕获内容可离线查看；索引重建不改变源数据；一万条对象下搜索交互仍保持可用。

### R2：资料驱动的 AI

目标：AI 的主要价值来自用户拥有的信息，而不是通用聊天。

- [x] 搜索结果和打开标签可加入最多 8 项的可见上下文托盘；普通发送不再隐式外发活动标签正文。
- [x] 多资料可在一次请求中按逐项 consent 读取，并受单项/总字符预算约束。
- [x] 带托盘资料的回答展示实际注入来源，点击回到原对象或浏览器 URL。
- [x] 比较、总结、时间线和研究报告的专用工作流与模板；运行前校验显式上下文数量。
- [x] AI 回答经可编辑预览和用户确认一键写入带来源引用的普通笔记。
- [x] AI 结果经目标选择和预览写入独立工作区任务，或写入本次实际引用的书签描述。
- [x] 可选本地 embedding/语义索引：用户显式下载固定版本的多语言 E5 q8 模型，离线 Worker 生成 384 维向量，并按“标题 > 正文 > 语义补充”与全文结果混排。
- [x] AI 写笔记提供写前预览、committed Node 回执及对话内产物记录。
- [x] 任务创建和书签描述更新提供版本绑定撤销；目标后续变化时由 Storage CAS 拒绝覆盖。
- [x] Agent 通用写工具的版本绑定脱敏预览与跨会话本机审计 outbox：本地 Node mutation 审批前绑定真实 provider version，副作用前先落 pending，明确回执后原子结算。

验收：用户能够选择三项本地资料生成带来源的新笔记，且未选择的私密正文不会进入模型请求。

### R3：可行动的连接

目标：从个人资料出发调用应用和服务，而不是堆连接入口。

- [x] 官方签名扩展包的固定目录发现、Minisign/摘要复验、持久 consent、权限查看、拒绝诊断与撤销；
- [x] HTTP/SSE/stdio MCP 的一致连接健康、脱敏失败、活动会话与最近调用诊断；
- [x] 应用和第三方 FileSystem 贡献可搜索资源与动作；
- [x] 社区发布从本地笔记/收藏生成草稿；
- [x] 外部 Agent 作为可选执行后端进入稳定链路。
- [x] 固定官方联网扩展 Registry 客户端：签名分页目录、可信缓存、离线/过期状态和手动签名包获取入口；安装仍独立复验与授权。
- [ ] 官方服务发布生产签名 feed，并完成真实分页、过期与离线缓存验收；当前固定端点返回 HTTP 404。
- [x] 第三方 publisher 双签名计划轮换：信任库迁移、单调密钥序列、永久退役指纹、旧包停用和确认后重新复验。

验收：连接的能力、数据范围和最近调用对用户可见；卸载连接不破坏已保存的本地对象。

### R4：跨端捕获与回顾

目标：移动端成为随时可用的个人信息入口。

- Android/iOS 分享扩展；
- 快速保存链接、文本、图片和文件；
- 移动端收件箱、搜索、阅读和轻编辑；
- 端到端加密同步的范围和冲突状态可见；
- 每日/每周回顾和未处理内容提醒。

验收：移动端离线捕获在恢复网络后可加密同步到桌面，重复提交不会生成重复对象。

## 8. 版本指标

### 8.1 北极星指标

**每周被重新利用的个人对象数**：本周被搜索打开、加入 AI 上下文、引用到笔记或执行动作的已保存对象数量。

该指标同时要求捕获、检索和再利用成立，比单纯保存数或对话数更接近长期价值。

### 8.2 漏斗指标

- 浏览页面 → 保存转化率；
- 捕获完成耗时与失败率；
- 保存对象 7/30 天内重访率；
- 有结果搜索率、结果点击率、查询改写率；
- 使用本地来源的 AI 会话占比；
- AI 结果转为文件/任务的比例；
- 同步成功率、冲突率和恢复率。

### 8.3 护栏指标

- 未经明确动作发送私密正文的次数必须为零；
- 捕获或索引导致的源数据丢失必须为零；
- 冷启动、搜索延迟和 bundle 预算不能因新增能力持续退化；
- 后端不可用时本地捕获、搜索、编辑和历史浏览继续工作。

## 9. 工程实施原则

- 新体验优先经过 FileSystem registry，不从 UI 直接绕到 IndexedDB。
- 轻量捕获复用 bookmark Node；正文快照先编码为带稳定来源标记的普通笔记，出现独立保留期或本机专属需求后再扩展模型。
- 搜索第一阶段允许查询时批量读取；达到规模门槛后迁移为可重建的本地派生索引。
- 每个里程碑提供聚焦单元测试、静态导出冒烟和真实 Tauri 验收步骤。
- 新连接必须同时定义信任来源、权限、撤销、导出、卸载和离线降级。
- 音频、Git、数据库和 Shell 作为工作区增强独立演进，不阻塞核心信息循环。

## 10. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 定位过宽 | 用户不知道为何首次打开 | 首屏和文案只讲浏览、捕获、查找、AI 再利用 |
| 查询时扫描正文 | 数据量增长后搜索变慢 | 设规模遥测阈值，R1 迁移本地增量索引 |
| 网页快照带来版权与存储压力 | 同步、导出和发布边界复杂 | 单页正文限制 80,000 字符，沿用笔记同步/导出边界；只有用户主动生成有界草稿、编辑预览并再次确认公开后才发送；后续提供清理策略 |
| AI 权限过强 | 私密数据外发或误写 | 保持正文 consent、工具 allowlist、写操作回执和高影响确认 |
| 连接能力稀释核心 | 工程投入分散 | 连接必须服务核心循环并通过利用率门槛 |
| 移动端完整复刻桌面 | 交付慢且体验臃肿 | 先做分享捕获、搜索、阅读和轻编辑 |

## 11. 当前执行记录

已交付：

1. R0 浏览器当前页一键收藏：读取真实标题和正文摘要，经统一书签 FileSystem 写入，并按 canonical URL 去重。
2. R0 全局命令面板从“仅名称”升级为跨对象内容匹配，并显示正文或元数据命中上下文。
3. R1 浏览器正文离线快照：以普通笔记持久化来源、捕获时间和有界正文，自动进入现有全文搜索。
4. R1 快照阅读状态条：明确显示离线状态、捕获时间，并可回到原始页面。
5. R1 选中文本摘录：桌面三种浏览器后端统一读取当前选区，生成带引用的可编辑摘录笔记。
6. R1 捕获收件箱：新书签、网页快照和摘录以标签派生为一个待整理列表，并支持无损归档。
7. R1 统一文件导入：浏览器书签 HTML 展开为去重书签；普通 HTML、PDF 和图片经 FileSystem 保存为收件箱资源。
8. R1 系统投递：桌面深链接、第二实例参数、“打开方式”和拖放由原生有界队列接收；URL 生成去重书签，文件复用统一导入。移动端已注册同一 URL scheme，原生文件分享扩展留待平台工程初始化。
9. R1 本地增量索引：IndexedDB v18 增加可丢弃派生仓，启动时迁移/重建，精确节点事件增量更新；异常时搜索回退源 FileSystem，设置页可查看状态并手动重建。
10. R2 可见上下文托盘：搜索结果和打开标签显式加入会话内引用；发送时逐项读取并限制正文预算，未选择的活动标签正文不再自动进入请求；回答保存来源引用并支持回跳。
11. R2 研究与产物闭环：提供资料总结、比较、时间线和研究报告模板；AI 回答先展示可编辑预览，确认后经 FilesPort 创建带“AI 生成”标签和来源引用的普通笔记，对话保存 committed Node 产物回执并可回跳。
12. R2 行动化与安全撤销：AI 回答可复制为所选工作区的独立任务线程；书签描述只允许写入本次回答实际引用的目标。两类操作都保存 committed version，撤销在 Storage 同一事务校验版本，目标被继续编辑后拒绝覆盖。
13. R2 统一工具预览与审计：本地文件写入、标签操作、浏览器控制、社区与关注等变更共用结构化目标/风险预览；本地 Node mutation 在审批前读取真实 metadata/version，模型不能自报 CAS 基线；表单值、正文、URL query 与凭据不进入预览、对话工具回执或审计。IndexedDB v19 以副作用前 pending + 明确回执后原子结算保存最多 1,000 条脱敏记录，崩溃后未确认结果不会消失，并以只读 `app.agent-write-audit` FileSystem 在“活动 / 审计”跨会话查看。
14. R2 可选本地语义检索：Transformers.js Worker 使用固定 commit 的 `Xenova/multilingual-e5-small` q8 ONNX 模型；模型约 129 MiB，只有用户在设置中确认后才下载，可独立删除。IndexedDB v20 的 384 维派生向量最多覆盖 10,000 个对象（约 14.7 MiB），源版本不一致、模型/Worker 异常或索引损坏时统一搜索自动回退全文结果。模型编排器与运行时随 App 打包并独立于 Next 主 chunk，生产门禁分别约束编排器、Worker 与 WASM 预算。
15. R3 官方签名扩展宿主：桌面端从固定 App 数据目录有界发现严格 manifest，以官方 Minisign 根和 connector SHA-256 在发现、授权、启动三处复验；只启动清理环境的进程外 MCP stdio connector，不向 webview 加载包代码。拒绝原因进入设置只读诊断，Catalog 继续要求系统凭据库 consent，并在撤销/卸载时终止 connector。
16. R3 MCP 一致诊断：stdio、SSE 与 Streamable HTTP 共用有界内存状态机，记录配置版本绑定的检测/健康/活动会话、连接耗时、工具数量和最近一次工具名/结果/耗时；连接、调用和关闭统一超时。原始 URL、命令参数、请求头、工具参数、返回正文和底层错误不会进入设置或对话事件，只输出稳定失败分类与错误码。设置页可实时查看并手动复测；配置 `updatedAt` 改变后旧健康状态立即失效。
17. R3 connector FileSystem 映射：已授权签名 connector 按 `resources:read` / `tools:invoke` 获得独立只读第三方 FileSystem；MCP resource/tool 的 URI 与原名只保留在活动会话内，稳定 FileRef 使用 SHA-256 不透明标识。全局搜索实时读取挂载中的标题、描述、类型和来源，不索引 URI 或正文。工具投影为带版本、风险和有界输入 schema 的 FileAction；调用前必须写入脱敏 pending 审计，写入失败阻止副作用，明确结果再结算为 committed/failed，参数、返回正文和远端错误不进入审计。
18. R3 社区发布草稿：本地笔记、书签和浏览捕获可生成普通 Note Node 草稿，来源内容按生成时快照复制，正文最多 20,000 字符；草稿未登录也可编辑、同步、搜索和进入回收站。预览与二次确认后只向 ServerPort 发送标题、HTTP(S) 链接和正文，来源 ID/版本不出站。公开发布与删除先写脱敏 pending 审计；断网、5xx、成功响应解析失败或 transport 异常一律保留“结果待确认”，发布页从审计 FileSystem 跨会话恢复重复提交锁。明确成功后把草稿归档为普通“社区发布”笔记；若归档失败，只允许重试本地归档，不会再次调用服务器。
19. R3 第三方签名扩展管理：本机 publisher 根按 ID/公钥指纹独立持久化并可撤销；publisher 签名撤销清单要求单调序列、字典序摘要和累积超集。原生文件选择器导入有界 `.ideall-extension` envelope，Rust 在写盘前完成验签、摘要与撤销检查；更新只前进版本并保留单版本回滚。安装、更新、回滚、根撤销和物理卸载经 Settings FileSystem specialized action 协调 Catalog teardown/consent 撤销，新代码不会继承旧授权。
20. R3 外部 Agent 执行后端：桌面端可在内置模型与用户配置的外部 ACP CLI Agent 间切换；initialize/session/prompt、流式文本、工具状态、取消、超时和诊断进入同一对话 UI。普通聊天拒绝权限，智能体模式逐次确认并先写 pending 审计；命令和 argv 只来自设备设置且不经 shell，也不会向外部进程注入 ideall MCP。
21. R0 统一快速捕获：资讯与社区嵌入页、聚合关注流、单来源详情、普通工具外链和内嵌浏览器共用 bookmarks FileSystem 的 `capture.bookmark` specialized action。标题、URL、摘要和 favicon 在 provider 边界有界规范化，新对象自动进入收件箱；canonical URL 查询与创建位于同一 IndexedDB 事务，跨入口、跨窗口和锚点变体不会重复。所有入口共用“保存到我的”的加载、成功与 existing 回执，保存结果继续进入全文索引和 AI 上下文链路。
22. R0 首次捕获引导：首个新建书签、网页快照、摘录、导入文件或系统分享成功后，以一次性 toast 动作回流到“我的 / 收件箱”；收件箱用非模态内联说明解释“保存 → 检查 → 归档”，完成一次归档或主动关闭后结束。状态只存本机公开配置，跨窗口同步；已有对象和重复捕获不启动或重置引导，配置不可用也不能阻断捕获。
23. R3 联网扩展 Registry 客户端：Rust 只访问固定官方 HTTPS 端点，不跟随重定向，逐页验证官方 Minisign 信封的原始 payload，并绑定序列、生成/过期时间、请求游标、全局排序、页数和条目上限。原始签名页原子缓存且每次读取重新验签；断网回退可信缓存并显示 stale/故障码。设置页刷新经 Settings FileSystem specialized action，目录只提供发现元数据和签名包地址，安装仍走既有 envelope、publisher、撤销、摘要和 consent 链。客户端门禁与首次签名 Release 已通过；固定 Wonita 端点当前仍返回 HTTP 404，等待生产部署已合并的代理路由。
24. R3 publisher 计划密钥轮换：严格信封的原始 payload 同时要求当前密钥授权和下一密钥持有证明；publisher 密钥序列只能加一，签名发布时间单调，最多 32 个历史指纹永久退役且不能回滚。信任库从 schema v1 读时迁移到 v2；设置页先检查候选并展示新旧指纹，确认时再次原生复验，然后先停止旧 connector、撤销 consent 并切换新根。旧密钥签名包及回滚副本立即失效，必须重新签名、安装和授权；已泄露当前密钥的独立恢复根仍未实现。
25. R3 签名扩展更新事务：只对已安装且 Registry 版本更高的包开放用户触发更新。Rust 从重新验签且未过期的缓存取 URL，解析 DNS 后拒绝任一非全局地址并钉连已检查 IP，请求禁代理、重定向和压缩；响应有界并校验文件 SHA-256。临时候选继续绑定 Registry 序列/有效期、publisher 指纹、manifest 摘要、版本与权限差异；确认时全部重验，桌面层先停止旧 connector、撤销 consent，再原子安装并保留单版本回滚。取消、成功和失败均清理临时包；不做后台静默下载或自动授权。
26. R3 生产 Registry 发布链：目录真相源经严格脚本绑定包、manifest、权限、publisher 指纹和 SHA，复用 App updater Minisign Secret 生成最多八页的短期信封；发布先上传并复核 staging Release，再原子切换固定通道，且拒绝 sequence 回退。Wonita apiserver 只从固定 HTTPS Release 代理有界 JSON 并短时缓存，不持有私钥；桌面再拒绝低序列或同序列分叉。首次空目录 Release 已发布并独立验签，跨仓服务代码已合并；剩余生产部署和真实端点验收。
