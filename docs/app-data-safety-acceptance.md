# Tauri App 数据安全验收

本手册用于在真实 Tauri 桌面进程和操作系统凭据库上验收本地数据链路。单元测试、静态导出冒烟和 `cargo test` 不能替代本验收；验收应在准备发布的安装包上至少覆盖一个主力平台，并在发布矩阵的 macOS、Windows、Linux 上周期性覆盖。

> 使用一次性测试工作区。归档导入会替换当前工作区，不要拿真实资料做破坏性验证，也不要把测试口令、同步码或系统凭据库内容写入日志、截图和 issue。

## 1. 系统凭据库往返

1. 启动安装后的 App；开发排障可用 `pnpm app:dev`，但浏览器里的 `pnpm dev` 不算本项验收。
2. 打开“设置 → 本地数据”，确认状态显示“敏感凭据由系统凭据库保护”。
3. 点击“运行系统凭据库自检”。
4. 通过标准：界面报告“写入、读回与清理自检通过”；系统凭据库中没有残留的 `ideall:self-test:*` 项。
5. 关闭系统凭据库服务或拒绝访问后重试。通过标准：App 明确报告失败，新凭据不会降级写入 `localStorage`；恢复服务后自检可再次通过。

自检命令只在 Rust 端生成一次性随机 key/value，不接收或返回用户凭据，并在读回后立即删除。它验证的是当前 App 进程到真实 keychain 的完整写、读、删链路。

## 2. 加密工作区归档

1. 在一次性工作区创建一个具有可识别标题的小文件或笔记。
2. 在“设置 → 本地数据”输入不少于 12 个字符的导出口令并导出。
3. 用文本编辑器打开 JSON。通过标准：外层 `kind` 为 `ideall.workspace-archive-encrypted`，文件中搜不到测试标题或正文。
4. 重新选择该归档。通过标准：未输入口令时只提示输入口令；错误口令提示“口令错误或加密归档已损坏”；正确口令才展示节点、Blob、回收站、插件和标签页数量。
5. 确认导入并重启 App。通过标准：数据与标签布局恢复，登录令牌、同步码和 API Key 不随归档迁移。
6. 选择超过界面标注上限的文件。通过标准：读取和 JSON/Base64 解码前即拒绝，不出现长时间无响应或进程异常退出。

口令不保存且无法找回。归档当前采用 PBKDF2-SHA-256（600,000 次）派生 AES-256-GCM 密钥；GCM 同时校验密文和外层归档元数据。

## 3. 本地持久化与导航映射

1. 在文件、音频和开发区各打开一个标签页，改变左侧活动栏、二级侧栏折叠状态和当前标签。
2. 完全退出 App 后重新打开。通过标准：IndexedDB 中的节点/Blob/回收站数量保持一致，工作区标签和活动模块按上次状态恢复。
3. 在“设置 → 本地数据”确认数据库健康、计数合理；WebView 支持时请求持久存储并确认状态更新。
4. 从活动栏切换模块并从二级侧栏打开条目。通过标准：模块导航只选择 provider/root，条目最终解析到真实 `FileRef`，不存在仅凭展示标签恢复数据的路径。详细映射见 [App 数据与导航映射](app-data-navigation.md)。

## 4. 运行时扩展授权与撤销

> 本项需要发行流水线提供的已签名测试 package、对应平台 verifier 和无真实用户数据的测试 connector；仓库不会用前端摘要模拟签名验证。

1. 在没有配置 verifier 的构建中发现测试 package。通过标准：设置页可显示其来源与权限，但“验证并授权”明确失败，factory/connector 不执行。
2. 使用正式 verifier 重新启动，点击“验证并授权”。确认界面先列出完整权限；通过标准：显示 verifier id/验证时间，系统凭据库新增 `ideall:runtime-extension-consent:*` 项，公开 `ideall:runtime-extensions:v2` 仅含 receipt id 与 descriptor 摘要。
3. 完全退出并重启。通过标准：来源重新验证、receipt 从系统凭据库恢复后才允许启用；篡改公开快照中的版本、内容摘要、权限摘要或 receipt id 均不能恢复授权。
4. 启用后撤销。通过标准：connector 先收到 abort 并停止，FileSystem/Engine 贡献消失，随后系统凭据删除；重启后不能恢复。
5. 再次授权并临时拒绝系统凭据库删除，然后撤销。通过标准：运行时仍立即停止，状态显示“撤销待重试”，不能启用或普通卸载；恢复凭据库后重试撤销，凭据删除且状态收敛为“已撤销”。
6. 在授权请求尚未返回时卸载来源。通过标准：迟到 receipt 被立即撤销，不生成可重放安装记录。

## 5. 社区发布草稿与未知结果保护

1. 未登录时打开“我的 → 发布”，分别从普通笔记、书签和浏览捕获生成草稿。通过标准：草稿可编辑和重启恢复；浏览捕获正文不带内部捕获时间；来源改变后既有草稿不被静默改写。
2. 在“我的 → 文件”和全局搜索中确认草稿是带“社区草稿”标签的普通笔记。把草稿移入回收站再恢复。通过标准：无需专用数据库或恢复路径，恢复后可重新出现在草稿列表。
3. 编辑标题、链接和正文并查看公开预览。输入 `javascript:`、`data:` 或非法链接。通过标准：只能保存/发送 HTTP(S) 链接，标题不超过 200、链接不超过 2,048、正文不超过 20,000 字符。
4. 点击“公开发布”。通过标准：确认框明确说明标题、链接和正文将发送并公开；取消时无网络 mutation；确认后“活动 → 审计”先出现不含正文、URL、来源 id/版本的 pending 记录，服务器明确成功后结算并把本地草稿改标为“社区发布”。
5. 分别模拟审计库不可写、HTTP 422、断网/超时、HTTP 503、成功响应中断，以及“服务器成功但本地归档冲突”。通过标准：审计不可写时请求不发送；422 结算 failed 并可修正重试；其余不确定结果保留 pending、显示“结果待确认”，重启后仍从审计 FileSystem 恢复重复提交锁。服务器已明确成功但本地归档失败时只显示“完成本地归档”，不会再次调用服务器。公开内容删除遵循同一未知结果规则。

## 6. 验收记录

每次记录以下非敏感信息：

| 字段 | 示例 |
| --- | --- |
| App 版本/提交 | `0.1.0 / <commit>` |
| OS 与版本 | `Windows 11 24H2` |
| 安装形态 | 正式安装包 / `tauri dev` |
| 系统凭据库自检 | 通过 / 失败及非敏感错误码 |
| 加密归档错误口令/正确口令 | 通过 / 通过 |
| 重启后数据与导航恢复 | 通过 |
| 遗留 fallback/公开凭据计数 | 仅记录数量，不记录值 |
| 运行时扩展验证/重启恢复/撤销重试 | 通过 / 通过 / 通过 |
| 社区草稿本地恢复/出站确认/未知结果保护 | 通过 / 通过 / 通过 |

提交证据前检查截图与日志，确保不含口令、token、同步码、API Key、consent receipt、归档正文或本机敏感路径。
